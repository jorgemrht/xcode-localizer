# Xcode Localizer

Zero rule: when the user asks for translations, screen text, placeholders, buttons, labels, or localized copy, do not locate the screen, do not search UI files, do not read Swift files, and do not read the Xcode project. The first action is only to check whether `Translations/` exists.

If `Translations/` does not exist, respond only:

```text
Translations folder does not exist. Where do you want me to create it? Recommended: ./Translations
```

Do nothing else until the user answers.

This skill has one job: create and update Xcode 26 String Catalog localization artifacts.

String Catalogs were introduced in Xcode 15, but this skill targets Xcode 26 only. Do not add compatibility paths for older Xcode versions.

## Only Outputs

The only persistent files this skill may create or modify are:

```text
Translations/Localizable.xcstrings
Translations/AppStrings.swift
Translations/reports/*.html
```

If `Translations/L10n.swift` exists and contains the `Auto-generated by xcode-localizer` header, delete it after writing `Translations/AppStrings.swift` to prevent stale generated APIs and duplicate symbols.

Do not create or modify anything else. In particular, never create or modify:

```text
*.swift outside Translations/
*.xcodeproj/**
project.pbxproj
Package.swift
Tuist files
JSON report files
.xcode-localizer.json
Translations/L10n.swift
.strings
.stringsdict
.xcloc
```

## Hard Rules

- Do not read, search, inspect, diff, open, create, modify, or delete app source files, UI files, project files, target membership, build settings, or manifests.
- Do not modify views or connect views to localization keys.
- Do not infer screen names from source files. Use only the user's words.
- If `Translations/` is missing, stop and ask where to create it before running any script.
- If `Translations/` exists but `Translations/Localizable.xcstrings` does not exist, create it.
- Generate an HTML review report every time localization text changes. The report must live under `Translations/reports/`.
- The HTML report table must use only `status`, `key`, and language columns. Do not include `warnings`, `screen`, `element`, or `meaning` columns.
- Do not generate JSON reports or config files.
- Do not integrate files into Xcode automatically.
- Do not use legacy localization formats.
- Preserve unknown fields in existing `.xcstrings` entries.
- Use `scripts/apply_xcstrings_changes.py` when a terminal is available.

## Swift API

Generate `Translations/AppStrings.swift` as a custom namespaced wrapper over `LocalizedStringResource`, grouped by screen.

Use this style:

```swift
AppStrings.Login.buttonSignIn
AppStrings.Common.buttonCancel
```

Do not use `NSLocalizedString`.

Do not write Xcode's generated `extension LocalizedStringResource` directly into the app target because Xcode can generate the same symbols automatically and cause invalid redeclarations.

## Key Convention

Use stable semantic keys:

```text
{screen}_{element}_{meaning}
```

Rules:

- `screen`: the screen name provided by the user. If missing, use `common`. Do not ask and do not inspect source files.
- `element`: one of `button`, `text`, `title`, `subtitle`, `placeholder`, `error`, `alert`, `accessibility_label`.
- `meaning`: short English semantic identifier.
- Use lowercase snake_case.

Examples:

```text
login_placeholder_username
login_placeholder_password
login_button_sign_in
login_button_remember_password
login_text_username
common_button_cancel
```

## Change Contract

Use this shape for the script input:

```json
{
  "sourceLanguage": "en",
  "items": [
    {
      "screen": "login",
      "element": "placeholder",
      "meaning": "username",
      "translations": {
        "en": "Username",
        "es": "Usuario",
        "it": "Nome utente"
      },
      "comment": "Placeholder for the username field on the login screen."
    }
  ]
}
```

The script derives the key from `screen`, `element`, and `meaning` unless `key` is explicitly provided.

For deletion:

```json
{
  "items": [
    {
      "key": "login_placeholder_username",
      "delete": true
    }
  ]
}
```

## Workflow

1. Check whether `Translations/` exists.
2. If `Translations/` does not exist, stop and ask where to create it. Do not run the script yet.
3. After the user confirms `Translations/`, create the folder and continue.
4. Create `Translations/Localizable.xcstrings` first if it does not exist.
5. Build the structured change JSON from the user request.
6. Run:

```bash
python3 /path/to/xcode-localizer/scripts/apply_xcstrings_changes.py apply --allow-create-translations --changes-json '<json>'
```

7. The script creates or updates only:

```text
Translations/Localizable.xcstrings
Translations/AppStrings.swift
Translations/reports/latest.html
Translations/reports/<timestamp>-l10n.html
```

8. Validate with Xcode tools when available:

```bash
xcrun xcstringstool print Translations/Localizable.xcstrings
xcrun xcstringstool compile Translations/Localizable.xcstrings --output-directory /tmp/xcode-localizer-check --dry-run
```

## Comments

Use the `.xcstrings` `comment` field for translator context. Xcode can generate comments automatically in the String Catalog editor. This skill may write explicit comments supplied or reasonably derived from the user's localization request, but it must not inspect app source to generate comments.
