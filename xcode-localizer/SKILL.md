# Xcode Localizer

One job: create and update Xcode 26 String Catalog localization artifacts inside `Translations/`. Never touch app source, SwiftUI views, Xcode project files, build settings, or legacy formats.

## Runtime Requirements

- Requires Python `3.14+` to run `scripts/apply_xcstrings_changes.py`.
- On macOS, install it with Homebrew:

```bash
brew install python@3.14
python3 --version
```

- For local tests, use a project virtual environment:

```bash
python3 -m venv .venv
.venv/bin/python -m pip install pytest
.venv/bin/python -m pytest tests/ -v
```

## Principles

| # | Rule | Why |
|---|------|-----|
| 1 | Check `Translations/` first, always | Prevents writing files to wrong locations |
| 2 | Never read app source or UI files | Agent must not drift into view editing when asked for copy |
| 3 | Stable `{screen}_{element}_{meaning}` keys | Renames break existing `LocalizedStringResource` references |
| 4 | All-language coverage on new keys | Partial catalogs cause runtime fallback to key names |
| 5 | `AppStrings.swift` wraps `LocalizedStringResource` | `NSLocalizedString` bypasses String Catalog validation |
| 6 | No `extension LocalizedStringResource` in app target | Xcode auto-generates the same symbols → duplicate declarations |
| 7 | HTML report on every change | Human review of what changed, when, and by whom |
| 8 | Config in `xcode-localizer.config.json` | Settings persist across sessions without re-asking the user |
| 9 | `extractionState: manual` on every entry | Prevents Xcode from treating agent-written strings as stale |
| 10 | No JSON report files | HTML is human-readable; JSON reports are noise in the repo |

## Zero Rule

When the user asks for translations, screen text, placeholders, buttons, labels, or localized copy:

1. Do not locate screens.
2. Do not read or search Swift files.
3. Do not read the Xcode project.
4. Check only whether `Translations/` exists.

If `Translations/` does not exist:

```text
Translations folder does not exist. Please tell me the path where I should create it.
```

Do nothing else until the user answers.

## Only Outputs

```text
Translations/Localizable.xcstrings
Translations/AppStrings.swift
Translations/xcode-localizer.config.json
Translations/reports/latest.html
Translations/reports/historyofchanges.html
Translations/reports/<dd-mm-yyyy-HHMMSS>-xcode-localizer.html
```

If `Translations/L10n.swift` contains `Auto-generated by xcode-localizer`, delete it after writing `AppStrings.swift`.

Never create or modify:

```text
*.swift outside Translations/   *.xcodeproj/**   project.pbxproj
Package.swift   Tuist files   .strings   .stringsdict   .xcloc
JSON report files   .xcode-localizer.json   Translations/L10n.swift
```

## Hard Rules

- Do not read, search, inspect, diff, open, create, modify, or delete app source files, UI files, project files, target membership, build settings, or manifests.
- Do not modify views or connect views to localization keys.
- Do not infer screen names from source files. Use only the user's words.
- If `Translations/` is missing, stop and ask where to create it before running any script.
- If `Translations/Localizable.xcstrings` does not exist, create it.
- Generate HTML review reports every time localization text changes.
- `latest.html` title: `Xcode Localizer - Current localizations`. Show `Last time generated: <dd-mm-yyyy hh:mm:ss>` under the page title. Includes views: all keys, key pattern groups, screen groups, element groups, and key descriptions. The `Go To Key descriptions` button and the full ungrouped Key descriptions section are visible only in the All keys view. Key descriptions are also shown inside each key pattern, screen, and element group.
- `historyofchanges.html` shows change reports from most recent to oldest. Show `Last time generated: <dd-mm-yyyy hh:mm:ss>` under the page title. Render one table with `date` and `author` columns plus a right-aligned review button column with no header text. Display dates as `dd-mm-yyyy hh:mm:ss`, not filenames. Include two filters next to `Current localizations`: a calendar date input and an author text input. If no rows match, hide the table and show centered text: `No localization text was generated for that date or author.`
- Change report columns: `status`, `key`, language old/new pairs (`en old`, `en`, `es old`, `es`…), `author`.
- Current report columns: `key`, language columns, `author`. No `warnings`, `screen`, `element`, or `meaning` columns.
- `author` resolves: `GIT_AUTHOR_NAME` → `git config user.name` → last commit author → `Unknown`.
- Do not generate JSON reports.
- Preserve unknown fields in existing `.xcstrings` entries.
- On a new key, provide translations for every language in config and every language already in the catalog.
- Write concise translator context in the top-level `comment` and concise per-language descriptions in each localization `comment`.
- Use `scripts/apply_xcstrings_changes.py` when a terminal is available.
- Do not use legacy localization formats.
- Do not integrate files into Xcode automatically.

## Key Convention

```text
{screen}_{element}_{meaning}
```

- `screen`: user-provided, snake_case. Default: `common`. Never infer from source.
- `element`: one of the configured `validElements` (see Default Settings).
- `meaning`: short English semantic identifier.
- All lowercase snake_case.

Good keys: `login_placeholder_username`, `login_button_sign_in`, `common_button_cancel`, `settings_title_notifications`.

Bad keys: `screen_btn_login`, `login_button_button`, `login_text_iniciar_sesion`, `login_label_1`.

## Swift API

Generate `Translations/AppStrings.swift` as a `LocalizedStringResource` wrapper grouped by screen:

```swift
AppStrings.Login.buttonSignIn
AppStrings.Common.buttonCancel
```

Do not use `NSLocalizedString`. Do not write `extension LocalizedStringResource` into the app target.

## Default Settings

```json
{
  "defaultLanguage": "en",
  "languages": ["en", "es"],
  "defaultScreen": "common",
  "validElements": ["accessibility_hint","accessibility_label","alert","button",
    "context_menu","empty_state","error","label","link","message",
    "navigation_title","picker","placeholder","subtitle","success",
    "tab","text","title","toggle","toolbar","warning"],
  "keyPattern": "{screen}_{element}_{meaning}",
  "swiftApiName": "AppStrings"
}
```

Create `Translations/xcode-localizer.config.json` when missing. Modify it only when explicitly requested.

## Change Contract

```json
{
  "defaultLanguage": "en",
  "items": [
    {
      "screen": "login",
      "element": "placeholder",
      "meaning": "username",
      "translations": { "en": "Username", "es": "Usuario", "it": "Nome utente" },
      "comments": {
        "en": "Login username placeholder.",
        "es": "Placeholder del usuario en login.",
        "it": "Placeholder nome utente nel login."
      }
    }
  ]
}
```

For deletion:

```json
{ "items": [{ "key": "login_placeholder_username", "delete": true }] }
```

## Workflow

1. Check whether `Translations/` exists. If not, ask where to create it and stop.
2. After confirmation, create `Translations/` and `Translations/Localizable.xcstrings` if needed.
3. Build the change JSON from the user request.
4. Run:

```bash
python3 /path/to/xcode-localizer/scripts/apply_xcstrings_changes.py apply \
  --allow-create-translations \
  --changes-json '<json>'
```

5. Validate when Xcode tools are available:

```bash
xcrun xcstringstool print Translations/Localizable.xcstrings
xcrun xcstringstool compile Translations/Localizable.xcstrings \
  --output-directory /tmp/xcode-localizer-check --dry-run
```

## Comments

Use the `.xcstrings` `comment` field for translator context. Write explicit comments when supplied or reasonably derived from the user request. Do not inspect app source to generate comments.
